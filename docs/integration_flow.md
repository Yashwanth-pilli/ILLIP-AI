# Frontend And Backend Integration Flow

The frontend is a React SPA (built with Vite) in `frontend/src/`, and the backend is a FastAPI app that returns JSON. The backend serves the built frontend directly — no separate frontend server needed.

## Local Services

| Service | Default URL | Source |
| --- | --- | --- |
| App (backend serves built frontend) | `http://127.0.0.1:8000/` | `frontend/dist/` (built from `frontend/src/`) |
| Frontend dev server (hot reload) | `http://localhost:3000` | `frontend/src/`, proxies `/api` to :8000 |
| Backend API | `http://127.0.0.1:8000/api` | `app/main.py`, `app/api/` |
| API docs | `http://127.0.0.1:8000/docs` | generated by FastAPI |

Build the frontend with `cd frontend && npm run build` (outputs to `frontend/dist/`, which `app/main.py` mounts and serves at `/`). For active frontend development, `npm run dev` runs a Vite dev server on :3000 with hot reload, proxying API calls to :8000.

## Request Path

```text
Browser
  -> frontend/src/api.js
  -> fetch("http://127.0.0.1:8000/api/...")
  -> app/main.py
  -> app/api/__init__.py
  -> app/api/routes/<feature>.py
  -> app/services/<feature>_service.py
  -> agents, providers, SQLite, or JSON files
  -> JSON response
  -> frontend updates the page
```

## Frontend Calls

`frontend/src/api.js` is the main integration file; `frontend/src/App.jsx` is the root component.

| UI action or refresh | Backend endpoint |
| --- | --- |
| Send chat message | `POST /api/chat/` |
| Health indicator | `GET /api/health` |
| System status panel | `GET /api/system/status` |
| Agent list panel | `GET /api/agents/` |
| Task stats | `GET /api/tasks/stats/overview` |
| Memory stats | `GET /api/memory/stats/overview` |

The frontend uses `textContent` for chat messages so user-entered chat text is rendered as text, not HTML. Dashboard panels use fixed backend fields from trusted local endpoints.

## Backend Route Map

Routes are collected in `app/api/__init__.py` under the `/api` prefix.

| Route module | Prefix | Purpose |
| --- | --- | --- |
| `health.py` | `/api/health` | Backend health |
| `chat.py` | `/api/chat` | Chat messages and history |
| `tasks.py` | `/api/tasks` | Task CRUD and stats |
| `memory.py` | `/api/memory` | Store, retrieve, and search memory |
| `agents.py` | `/api/agents` | Agent status and execution |
| `system.py` | `/api/system` | System status and info |

## Startup Order

Run setup once:

```powershell
.\scripts\setup.ps1
```

Build the frontend (only needed after `frontend/src/` changes):

```powershell
cd frontend
npm install
npm run build
cd ..
```

Start the backend — it serves the built frontend automatically:

```powershell
.\scripts\run_backend.ps1
```

Open:

```text
http://127.0.0.1:8000/
```

## Single-Server Design

`app/main.py` mounts `frontend/dist/` (the Vite build output) when it exists, falling back to `frontend/` otherwise. The backend serves both the API and the UI from `http://127.0.0.1:8000/` — no separate frontend server in production.

For active UI development, run `npm run dev` inside `frontend/` for a hot-reloading dev server on `:3000` that proxies `/api` and `/data` to `:8000`.

## Keeping Integration In Sync

When changing integration behavior:

1. Update the backend route or response schema.
2. Update `frontend/src/api.js` to call the new route or field.
3. Update tests for the backend contract.
4. Rebuild the frontend (`npm run build`) so `frontend/dist/` matches.
5. Update this file and `docs/api.md` if the public endpoint changed.

This keeps the beginner path reliable: read the docs, open the frontend, inspect the API docs, and see the same route names everywhere.
